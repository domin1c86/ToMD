# Design Language Extractor 产品设计

## 1. 产品定义

Design Language Extractor 是一款面向个人开发者的本地桌面应用。用户上传网站或 App 的多张参考截图，应用调用用户自行配置的多模态 AI API，提取其中可迁移的设计语言。用户校正分析结果后，应用导出通用 `DESIGN.md`，供 Codex、Claude Code、Cursor 等 AI 编程工具在开发新产品时遵守。

产品不是截图转代码或界面复刻工具。它提取视觉规律和设计约束，同时排除原产品的品牌、文案、Logo、专有素材和业务内容。

### 成功标准

另一个 AI 在不查看原始截图的情况下，仅通过读取导出的 `DESIGN.md`，能够为新产品生成设计语言一致、规则遵守度高且没有照搬原品牌元素的界面。

## 2. 目标用户与 MVP 范围

首批用户是使用 AI 编程工具构建个人产品的个人开发者。

MVP 包含：

- 创建和管理本地项目。
- 为项目上传多张网站或 App 截图。
- 标注截图所属平台、页面和场景。
- 配置用户自己的多模态 AI API。
- 从截图生成结构化 `DesignSpec`。
- 人工确认、编辑、接受或驳回分析规则。
- 实时预览并导出通用 `DESIGN.md`。
- 保存本地分析版本和导出历史。

MVP 不包含：

- URL 自动抓取、Figma 导入或视频分析。
- 截图转代码和像素级界面复刻。
- 云同步、团队协作、审批和多人权限。
- 模板市场、社区分享和计费系统。
- IDE 插件、MCP、公共 API 和 GitHub Action；这些属于后续阶段。

## 3. 产品形态与核心流程

产品以 Tauri 桌面应用交付，界面使用 Web 技术实现。截图、项目数据、结构化分析结果、API 凭据引用和导出文件均保存在本机。

核心流程：

1. 用户创建项目，填写新产品类型和目标平台。
2. 用户上传多张参考截图，并标注页面与场景。
3. 应用在调用 API 前明确展示将发送的图片、供应商和模型。
4. 分析编排层处理图片并调用用户配置的多模态 API。
5. 模型返回符合统一 JSON Schema 的 `DesignSpec`。
6. 应用校验结果；格式错误时执行有限修复，仍失败则显示可诊断错误并允许重试。
7. 用户在工作台中查看证据和置信度，校正规则。
8. 确定性模板将已确认的 `DesignSpec` 编译为 `DESIGN.md`。
9. 用户导出 Markdown，或复制内容到自己的代码项目。

## 4. 工作台信息架构

MVP 只保留四类一级页面：

- 项目列表：创建、打开、重命名、归档和删除本地项目。
- 截图与场景管理：上传、预览、分类、排序和移除截图。
- 分析与校正工作台：审核结构化规则和证据。
- 导出历史：预览、复制、重新导出和比较历史版本。

分析与校正工作台采用三栏布局：

- 左栏显示截图、场景、当前证据和引用区域。
- 中栏以结构化表单编辑设计规则，不以聊天作为主要交互。
- 右栏实时预览最终 `DESIGN.md`。

规则编辑器按以下领域组织：

- 设计意图与视觉原则。
- 色彩角色和使用限制。
- 字体层级、字重、行高和文本密度。
- 间距尺度、布局网格和响应式原则。
- 圆角、边框、阴影和层级。
- 组件形态和组件状态。
- 图标、插图和图片风格。
- 动效与交互反馈。
- 应该做、不应该做和允许变化的范围。

每条规则显示：

- 可编辑的规则内容。
- 类型：观察事实、归纳规律或执行建议。
- 来源截图和证据说明。
- 置信度。
- 接受、修改或驳回状态。

## 5. DesignSpec 中间层

模型不能直接生成最终 Markdown。所有供应商输出必须先转换为版本化的 `DesignSpec`，再由本地模板生成 Markdown。

`DesignSpec` 的顶层结构包括：

- `metadata`：Schema 版本、项目平台、分析时间、模型和来源截图。
- `intent`：整体气质、信息密度、视觉层级和设计原则。
- `tokens`：颜色、字体、间距、尺寸、圆角、边框和阴影。
- `layout`：容器、网格、对齐、留白和响应式规律。
- `components`：组件类型、视觉特征、变体和状态。
- `assets`：图标、插图、摄影和背景素材方向。
- `motion`：动效目的、时长倾向和反馈方式。
- `constraints`：必须遵守、禁止使用和允许变化的规则。
- `evidence`：规则与截图之间的引用关系。
- `uncertainties`：证据不足、截图冲突和需要用户确认的项目。

所有可审核规则统一使用以下记录结构：

- `id`：项目内稳定且唯一的字符串 ID。
- `category`：所属规则领域。
- `statement`：供人和 AI 阅读的可执行规则。
- `kind`：`observation`、`pattern` 或 `recommendation`。
- `scope`：`global`、`platform`、`page_type` 或 `component`。
- `value`：可选结构化值，例如颜色、尺寸或 token 引用。
- `evidence_ids`：关联的证据记录 ID。
- `confidence`：0 到 1 的数值。
- `status`：`pending`、`accepted`、`edited` 或 `rejected`。
- `source`：`model` 或 `user`。

证据记录包含稳定 ID、截图 ID、可选区域坐标和简短说明。`metadata.schema_version` 首版固定为 `1.0`；读取器拒绝未知主版本，对新增可选字段保持兼容。该契约支持局部重新分析、版本比较以及后续 CLI、MCP 和 API 消费。

## 6. AI 分析策略

MVP 使用单一结构化分析管线，不使用多 Agent 编排：

1. 检查截图格式、尺寸和数量。
2. 根据模型限制压缩或分批组织图片。
3. 发送固定系统指令、项目上下文、截图场景信息和 JSON Schema。
4. 要求模型区分可见事实、跨截图规律和迁移建议。
5. 要求模型避免输出原品牌名、原文案、Logo 和专有素材。
6. 校验 Schema、字段范围和证据引用。
7. 对可修复的格式问题执行一次结构化修复。
8. 将冲突或低置信度规则标记给用户，而不是自动确定。

分析应优先识别跨多张截图重复出现的规律。只在单张截图出现的特征默认降低置信度，不应自动升级为全局规范。

## 7. 模型供应商与隐私

MVP 提供两种接入方式：

- 主流多模态 API 的预设适配器。
- OpenAI-compatible 自定义端点，可配置 Base URL、API Key 和模型名。

供应商适配层统一负责：

- 能力检测和配置验证。
- 图片输入和结构化输出的请求转换。
- 超时、限流、认证失败和模型不兼容错误的归一化。
- Token/图片数量限制提示。

隐私规则：

- API Key 仅保存到操作系统凭据库，不写入项目文件或日志。
- 截图、项目和分析结果默认只保存在本机。
- 只有用户主动开始分析时，选中的截图和必要提示才发送到其配置的 API。
- 调用前明确展示供应商、模型和发送图片数量。
- 日志不得记录 API Key、完整请求正文或图片内容。

## 8. Markdown 输出契约

导出的文件默认命名为 `DESIGN.md`，面向不同 AI 编程工具使用同一套通用格式。

固定章节：

1. Design intent
2. Visual principles
3. Design tokens
4. Typography
5. Layout and responsive rules
6. Component conventions
7. Interaction and states
8. Image and icon direction
9. Do / Don't
10. AI implementation checklist

输出原则：

- 使用明确、可执行的指令，避免仅使用“现代”“高级”等主观词。
- 同时描述适用条件和禁止用法。
- 使用语义角色，不依赖原产品组件名。
- 对无法可靠推断的内容明确标注，不编造精确值。
- 不包含原截图、品牌名、原始文案或专有资产。
- 保持足够紧凑，避免占用 AI 编程工具过多上下文。

## 9. 错误与边界处理

- 图片损坏或格式不支持：上传阶段阻止并说明可接受格式。
- 截图过少：允许分析，但提示结果可能只代表单页面特征。
- 截图互相冲突：分别列出证据并要求用户选择或限定适用场景。
- API Key 无效：保留项目状态，显示供应商返回的归一化认证错误。
- 模型不支持图片或结构化输出：配置测试阶段阻止使用。
- 请求超时或限流：允许安全重试，不重复创建分析版本。
- Schema 校验失败：执行一次修复；失败后保留原始诊断信息，但不将无效结果写入正式版本。
- 用户移除截图：标记依赖该截图的规则为需要重新验证，不静默删除用户编辑内容。

## 10. 验收与测试

### 功能测试

- 创建项目、上传多张截图、保存场景标注并重新打开。
- 配置预设供应商和 OpenAI-compatible 端点。
- 成功分析并生成有效 `DesignSpec`。
- 编辑、接受、驳回规则后，Markdown 预览同步更新。
- 重新分析局部内容时保留不相关的人工修改。
- 导出、复制和恢复历史版本。
- 离线状态下仍可打开和编辑已有项目。

### 安全与隐私测试

- 项目文件、日志和崩溃报告中不出现 API Key。
- 未点击分析时不产生外部网络请求。
- 调用请求只包含当前选中的截图。
- 删除项目后按用户确认清理对应本地截图和分析文件。

### 模型兼容测试

- 每个预设适配器使用同一组基准截图。
- 验证正常响应、非 JSON 响应、缺失字段、超时、限流和认证失败。
- 验证不同模型生成的结果都能被统一 Schema 和编辑器处理。

### 产品效果评估

建立固定基准集，包含网站、桌面 App 和移动 App 的多页面截图。每个测试由分析模型生成规范，再让另一个 AI 只读取 `DESIGN.md` 实现一个未出现过的新页面。

人工评分维度：

- 设计语言一致性。
- 明确规则遵守率。
- 新场景的可迁移性。
- 是否错误复制原品牌或内容。
- 用户校正所需时间。

MVP 达标条件是：用户无需查看原截图即可理解和修改规则，且只读取 `DESIGN.md` 的实现 AI 能稳定产出可辨识的一致风格。

## 11. 演进路线

### 阶段 A：本地工作台

完成截图管理、模型配置、结构化分析、人工校正、版本记录和 Markdown 导出，验证核心成功标准。

### 阶段 B：开发工具接入

提供 CLI 和本地 MCP Server，读取同一份 `DesignSpec`，为 Codex、Claude Code、Cursor 等工具按需提供设计规则和组件约束。

### 阶段 C：设计规范基础设施

开放本地 API、批处理、目录监听和自动化工作流。只有在个人用户明确需要跨设备或团队协作后，才增加可选云同步与团队治理。

## 12. 技术边界

- 桌面容器：Tauri。
- 前端：Web 技术栈，具体框架在实施计划中结合仓库现状确定。
- 本地数据：SQLite 保存结构化数据，文件系统保存截图和导出文件。
- 凭据：操作系统安全凭据库。
- 核心模块边界：项目存储、截图管理、供应商适配、分析编排、Schema 校验、规则编辑和 Markdown 编译相互独立。
- `DesignSpec` 和 Markdown 编译器不依赖桌面 UI，以便阶段 B、C 复用。
